# 07-02 Plan: Observability Reporting and Failure Diagnostics

## Objective
- Surface structured runtime metadata and job failure diagnostics through the dashboard/logging views so operators can understand what happened without reading raw free-text logs.

## Current State
- `07-01` captures structured metadata in package log `details`, but the dashboard only shows time, stage, status, and message.
- Job retry state exists in `ExtractionJob`, but the UI does not expose attempt count, last error, or queue state in an operator-friendly way.
- Failure diagnostics still require reading raw logs or direct DB inspection.

## Target State
- Dashboard log views expose structured metadata such as model id, prompt version, latency, token usage, and relevant failure details in a readable form.
- Failure and retry diagnostics show whether a package failed during ingestion, classification, extraction, or queue retry handling.
- Sensitive fields remain excluded or bounded so observability does not become a PII leak.

## Implementation Steps
1. Add helpers to parse and normalize structured `PackageLog.details` payloads for UI consumption.
2. Extend dashboard/log views to display runtime metadata and failure diagnostics in a compact operator-friendly format.
3. Surface relevant `ExtractionJob` retry/failure state for packages when present.
4. Add focused tests for metadata parsing, dashboard query behavior, and failure-state rendering.

## Risks
- Compatibility risk: low; this is a presentation/query layer change on top of existing logs.
- Migration risk: low; historical logs may have plain-text or null `details` and must degrade gracefully.
- Operational risk: medium; observability views must avoid exposing excessive raw payload content or sensitive values.

## Validation
- Tests to run:
  - `pytest tests/test_ui_queries.py tests/test_dashboard.py -q --basetemp=.pytest-lite-07-02`
- Manual verification:
  - inspect a successful package log and confirm model/prompt/latency fields appear
  - inspect a failed or retried package and confirm retry count and last error are visible
  - confirm logs remain readable when `details` is missing or plain text
- Success criteria:
  - runtime metadata is visible without reading raw JSON manually
  - retry/failure causes are easier to diagnose from the dashboard
  - logs degrade safely for older records without structured details
