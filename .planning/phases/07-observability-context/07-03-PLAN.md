# 07-03 Plan: Observability Analytics and Trend Reporting

## Objective
- Build aggregate observability views and dashboard reporting for latency, token usage, retries, and failure trends on top of the structured metadata captured in Phase 7.

## Current State
- Package-level logs now include structured runtime metadata and queue diagnostics.
- The dashboard can show per-package metadata, but there is no aggregate view across packages for latency, token usage, retry rates, or recent failures.
- Operators still need to inspect individual packages rather than seeing trend-level health signals.

## Target State
- `AnalyticalService` can derive aggregate observability metrics from structured log details and queue/job state.
- The dashboard shows compact trend/reporting widgets for recent failures, retry pressure, latency, and token usage where available.
- Missing or older unstructured logs degrade safely without breaking the aggregate reporting views.

## Implementation Steps
1. Add analytical queries/views over `package_logs.details` and extraction job state for aggregate observability metrics.
2. Extend dashboard reporting with summary metrics and compact tables/charts for recent failures and runtime trends.
3. Ensure unstructured or partially structured legacy log records are ignored or handled safely.
4. Add focused analytical and dashboard tests for aggregate metric calculations and UI rendering helpers.

## Risks
- Compatibility risk: low; this builds on existing structured log details and dashboard patterns.
- Migration risk: low; legacy logs without JSON details must be treated as partial data.
- Operational risk: medium; aggregate reporting must avoid surfacing sensitive detail payloads directly.

## Validation
- Tests to run:
  - `pytest tests/test_analytical.py tests/test_dashboard.py -q --basetemp=.pytest-lite-07-03`
- Manual verification:
  - inspect the dashboard after processing multiple packages
  - confirm aggregate failure counts, retry counts, latency, and token metrics are populated when data exists
  - confirm the UI remains stable when some logs have no structured details
- Success criteria:
  - dashboard exposes aggregate observability trends instead of only package-level diagnostics
  - analytical queries can summarize failures, retries, and latency/token usage
  - legacy/unstructured logs do not break the reporting layer
