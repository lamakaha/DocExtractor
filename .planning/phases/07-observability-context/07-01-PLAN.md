# 07-01 Plan: Observability and Package Context

## Objective
- Improve pipeline observability and reduce first-page classification bias by introducing explicit metadata capture and richer package-context-aware classification inputs.

## Current State
- Classification still relies on the first canonical PDF page rather than a richer package-level context.
- Pipeline logs record status transitions, but they do not capture model id, prompt/schema versions, latency, token usage, retry causes, or structured pipeline metadata.
- The original architecture review identified both observability and document-centric classification as remaining gaps after the canonical PDF refresh.

## Target State
- Classification uses a bounded package context that can incorporate multiple canonical pages and relevant extracted text instead of only the first page.
- Pipeline execution records structured metadata for classification and extraction runs, including model id, prompt/schema version, latency, and retry/failure context where available.
- Logging remains compatible with the current SQLite-backed pipeline while making later reporting and debugging materially easier.

## Implementation Steps
1. Define the minimum structured observability metadata to persist or log for classification/extraction runs.
2. Refactor classification input assembly so it can use a bounded package context instead of only page 1.
3. Update pipeline logging/metadata capture to include runtime details and failure causes in a queryable form.
4. Add focused tests for package-context classification selection and metadata/log emission.

## Risks
- Compatibility risk: medium; changes to classification inputs may alter document-type decisions.
- Migration risk: low; this slice extends logging and metadata rather than replacing core storage contracts.
- Operational risk: low to medium; richer prompts may slightly increase token and latency costs.

## Validation
- Tests to run:
  - `pytest tests/test_classification.py tests/test_extraction_pipeline.py -q --basetemp=.pytest-lite-07-01`
- Manual verification:
  - process a representative package with multiple pages or supporting text
  - confirm logs expose model/version/latency metadata
  - confirm classification no longer depends solely on the first page when additional package context is available
- Success criteria:
  - classification uses bounded package context
  - observability metadata is captured consistently
  - failure and retry causes are clearer than the current free-text-only flow
