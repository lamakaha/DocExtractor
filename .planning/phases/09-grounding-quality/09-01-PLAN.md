# 09-01 Plan: BBox Grounding Hardening

## Objective
- Improve grounding quality by tightening the extraction prompt, rendering higher-resolution page inputs for the model, and adding bbox sanity auditing so visibly suspect grounding can be identified in logs.

## Current State
- The pipeline now persists raw extraction responses and renders stored bboxes faithfully in the reviewer.
- Recent paydown tests show that some models still produce incorrect bbox coordinates even when the extracted values are correct.
- There is currently no prompt specialization for “tight value-only boxes,” no explicit extraction-render DPI control, and no bbox-quality audit beyond schema validation.

## Target State
- Extraction prompts emphasize exact value-only grounding and explicit fallback behavior when the model cannot localize a field confidently.
- Canonical PDF pages are rendered at a higher and configurable resolution for extraction input.
- Extraction audit details include bbox-quality warnings so operators can distinguish plausible grounding from suspicious model output.

## Implementation Steps
1. Tighten the extraction prompt around exact-value localization and fallback behavior for uncertain boxes.
2. Increase canonical PDF render resolution for extraction pages and make it configurable.
3. Add bbox-quality audit heuristics to extraction metadata and surface warnings in the extraction log details.
4. Add focused tests for prompt/version changes, render configuration, and bbox-audit output.

## Risks
- Compatibility risk: low; this is a prompt/input-quality improvement layered on the current pipeline.
- Operational risk: medium; higher render DPI increases processing cost.
- Quality risk: medium; bbox heuristics should inform debugging without silently corrupting otherwise-valid model output.

- Success criteria:

  - extraction page render DPI is higher/configurable
  - suspicious bbox patterns are recorded in audit metadata
