# 06-04 Plan: Config-Driven Analytical Shredding

## Objective
- Replace hardcoded analytical JSON shredding with config-driven field mappings so new document types can participate in DuckDB views and Excel export without code changes.

## Current State
- `AnalyticalService` hardcodes summary and transaction JSON paths for one specific extraction shape.
- `ExcelExporter` assumes a fixed summary/transactions contract derived from those views.
- Adding or changing document types currently requires Python code edits rather than configuration updates.

## Target State
- Document configs define which extracted fields map into analytical summary and transaction outputs.
- `AnalyticalService` builds DuckDB queries/views from config metadata instead of embedding document-specific JSON paths in code.
- Export continues to produce stable Summary and Transactions tabs, but the selected fields are sourced from config-driven mappings.
- Unsupported or partially configured document types fail visibly and safely instead of silently producing broken exports.

## Implementation Steps
1. Extend the document config schema with analytical/export mapping metadata for summary and repeating transaction fields.
2. Refactor `AnalyticalService` to load those mappings and build views or query fragments dynamically per document type.
3. Update export integration so the Summary and Transactions tabs consume the config-driven analytical output.
4. Add tests covering multiple document types, missing mappings, and stable export behavior.

## Risks
- Compatibility risk: medium; analytical queries and export consumers depend on current hardcoded column names.
- Migration risk: medium; existing configs will need analytical metadata before they participate in exports.
- Operational risk: low; this is primarily query-generation logic, but malformed configs could break analytical views.

## Validation
- Tests to run:
  - `pytest tests/test_analytical.py tests/test_export_service.py -q --basetemp=.pytest-lite-06-04`
- Manual verification:
  - approve at least two different document types
  - verify DuckDB summary output reflects config-selected fields
  - verify Excel export tabs remain valid and readable
- Success criteria:
  - new document types can opt into analytical shredding via config only
  - exports no longer depend on hardcoded JSON field names in Python
  - invalid/missing config mappings produce clear failures
