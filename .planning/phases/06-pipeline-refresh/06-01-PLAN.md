# 06-01 Plan: Canonical PDF Foundation

## Objective
- Introduce a canonical PDF normalization stage for the current extraction path and align bbox persistence with the reviewer contract.

## Current State
- The pipeline selects a primary file and processes PDFs/images/text differently.
- Non-PDF reviewer behavior is inconsistent because extraction records may point at images or text files directly.
- Bounding boxes are scaled to pixels before persistence even though the UI expects normalized boxes.

## Target State
- The extraction path normalizes the selected source document to a canonical PDF before classification and extraction.
- Extractions reference the canonical PDF artifact for review.
- Stored bounding boxes remain normalized in `0-1000` coordinates.

## Implementation Steps
1. Add a canonical document service for PDF passthrough and basic image/text/csv/html-to-PDF conversion.
2. Update the extraction pipeline to create or reuse a canonical PDF `ExtractedFile`.
3. Extract from rendered pages of the canonical PDF and persist normalized triplets.
4. Add focused tests for conversion and pipeline persistence behavior.

## Risks
- Compatibility risk: low to medium; reviewer should improve for non-PDF files but conversion fidelity may vary.
- Migration risk: low; original source files remain stored.
- Operational risk: medium; canonicalization adds CPU and memory usage.

## Validation
- Tests to run:
  - `pytest tests/test_canonical_document_service.py tests/test_extraction_pipeline.py`
- Manual verification:
  - ingest a text-like file and confirm review points to a PDF artifact
  - confirm bbox locate behavior still works
- Success criteria:
  - canonical PDF is created for supported non-PDF sources
  - extraction records link to the canonical PDF file
  - stored bboxes remain normalized
