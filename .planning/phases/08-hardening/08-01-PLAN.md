# 08-01 Plan: Runtime Hardening

## Objective
- Strengthen the durable job runtime so extraction work remains recoverable, diagnosable, and operationally safe when workers crash, claims go stale, or retries are exhausted.

## Current State
- Extraction work already uses persisted `ExtractionJob` records instead of inline watcher execution.
- Jobs can be enqueued, claimed, retried, completed, and failed with basic structured diagnostics.
- The current runtime still treats retries and failures with a relatively simple policy and does not provide a stronger stale-claim or terminal-failure model.

## Target State
- Job claiming behaves predictably when a worker dies or stops renewing progress.
- Stale or abandoned claims can be recovered without creating ambiguous double-processing states.
- Terminal failures are explicit and distinguishable from retryable failures.
- Operators and tests can reason about queue state transitions without relying on implicit side effects.

## Implementation Steps
1. Review the current `ExtractionJob` lifecycle and identify weak spots around stale claims, retry exhaustion, and idempotent recovery.
2. Introduce stronger runtime state handling for stale claims and explicit terminal failure behavior.
3. Update the watcher/worker path so claimed jobs are resumed or failed consistently rather than remaining stuck in ambiguous processing states.
4. Add focused tests for stale-claim recovery, bounded retries, and terminal job handling.

## Risks
- Compatibility risk: medium; queue semantics affect watcher and worker behavior.
- Migration risk: low; changes build on the existing persisted job model rather than replacing it.
- Operational risk: medium; incorrect recovery semantics could create duplicate processing or stuck jobs if not validated carefully.

## Validation
- Tests to run:
  - `pytest tests/test_extraction_job_service.py tests/test_watcher.py tests/test_extraction_pipeline.py -q --basetemp=.pytest-lite-08-01`
- Manual verification:
  - create or simulate a stuck processing job and confirm it becomes reclaimable or terminal in a predictable way
  - confirm retry exhaustion leaves a clearly inspectable terminal state
  - confirm successful jobs still complete without duplicate queue transitions
- Success criteria:
  - stale claims no longer leave jobs indefinitely stuck in `PROCESSING`
  - retry exhaustion results in an explicit terminal state
  - queue transitions remain idempotent and testable
