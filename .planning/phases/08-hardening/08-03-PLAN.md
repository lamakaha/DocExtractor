# 08-03 Plan: Reviewer BBox Debugging and Raw Response Audit

## Objective
- Determine whether observed bbox failures come from the model output, response parsing, or reviewer rendering, and make the pipeline persist enough evidence to answer that question directly for future runs.

## Current State
- The pipeline stores processed extraction JSON with normalized bounding boxes but does not persist the raw model response that produced those fields.
- Debugging a bad bbox currently requires inferring from post-processed payloads and UI behavior after the fact.
- The reviewer uses `streamlit-drawable-canvas`, which may be contributing additional rendering error or coordinate distortion.

## Target State
- Extraction runs retain raw model output or equivalent audit data so bbox debugging can compare model output against stored triplets directly.
- Reviewer bbox rendering is validated against the canonical PDF display path and corrected if the current canvas layer is distorting otherwise-valid boxes.
- Operators can tell whether a bad bbox came from the model, parsing, or rendering without reverse-engineering the pipeline.

## Implementation Steps
1. Persist raw extraction response or equivalent audit payload alongside processed extraction results.
2. Add pipeline- or UI-level helpers/tests that verify stored normalized boxes render correctly on the canonical PDF page.
3. Simplify or replace the reviewer highlight path if `streamlit-drawable-canvas` is introducing bbox drift.
4. Validate against at least one known-good and one known-bad paydown extraction case.

## Risks
- Compatibility risk: low to medium; changing reviewer rendering may affect the current approval workflow.
- Migration risk: low; this is primarily additive debug/audit data and UI rendering correction.
- Operational risk: medium; raw-response storage must remain bounded so the database does not accumulate excessive payload size.

## Validation
- Tests to run:
  - `pytest tests/test_extraction_pipeline.py tests/test_ui_form.py tests/test_ui_approval.py -q --basetemp=.pytest-lite-08-03`
- Manual verification:
  - run at least one paydown extraction from the manual fixtures
  - inspect the stored raw response for that extraction
  - compare the stored normalized bbox to the reviewer overlay on the canonical PDF page
  - confirm the diagnosis path is clear when a model returns visibly bad grounding
- Success criteria:
  - raw model output is available for extraction audit
  - reviewer overlays match stored normalized coordinates on the canonical PDF
  - future bbox issues can be attributed to model, parsing, or rendering with evidence
